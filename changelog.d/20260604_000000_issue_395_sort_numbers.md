### Added
- Issue #395: a concrete "sort these numbers in <language>, give me the code and the result" request now routes to `write_program` instead of `unknown`. Rather than a narrow sorting handler, this ships a universal, data-oriented numeric-list coding engine (`src/solver_handlers/numeric_list/`): it reads the operation, the given numbers, and the target language from the prompt via the seed ontology, then generates idiomatic code in the requested programming language (JavaScript, TypeScript, Python, Rust, Go, Ruby, Java, C#, C, C++) and — because every numeric-list operation is a pure, total function — computes and shows the result deterministically in the solver itself (no runtime needed).
- The engine covers seven operations out of one shared algorithm — `sort`, descending sort (`reverse_sort`), `reverse`, `sum`, `product`, `minimum`, `maximum` — driven by the `data/seed/meanings-numeric-list.lino` ontology (each operation maps to a family of list_transformation/list_reduction and a result_kind of list/scalar). Operations, language aliases, and localized answer text are all data, not hard-coded constants, so new operations and languages extend the seed rather than the code. Reductions are gated behind a `code_request` signal to avoid over-matching prose. Localized rendering covers all four supported UI languages (English, Russian, Hindi, Chinese).

### Changed
- Mirrored the numeric-list engine in the web runtime (`src/web/formal_ai_worker.js`, `tryNumericList`) so the browser and Rust paths produce byte-identical code and results, verified by `experiments/issue-395-js-numeric-list.mjs`.
- Added `examples/numeric_list_execution.rs`, an execution-verification harness that compiles and runs every generated program across the available toolchains and asserts the program's stdout equals the solver's computed result, closing the loop between the "verified by construction" claim and real execution.
