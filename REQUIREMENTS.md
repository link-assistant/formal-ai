## Current Scope Boundary

This PR is a proof of concept, not a full universal problem solver. The implemented core proves that the repository can expose OpenAI-shaped interfaces while returning symbolic, inspectable, Links Notation-backed answers and reviewable seed datasets. Full imports of very large corpora such as Wikipedia, Wikidata, Rosetta Code, Wikifunctions, and hive-mind history should run as chunked follow-up dataset jobs behind the same `.lino` format and 1500-line file limit.

## Issue #18 Full-Memory Export/Import Requirements

Issue [#18](https://github.com/link-assistant/formal-ai/issues/18) reports that
the **Export memory** button on the deployed demo saves only the in-session
event log, not the full agent state. The requirements below make full-memory
exports the universal default on every surface and surface known data
migrations on import.

| ID | Requirement | Status |
| --- | --- | --- |
| R109 | The default **Export memory** action must always produce the full, self-contained agent state (seed + memory log + UI preferences + environment metadata + version). No second button click is required to get the full export. | Implemented by `FormalAiMemory.exportFullMemory({ seed, events, preferences, info })` in `src/web/memory.js` (wired to the topbar `Export memory` button via `handleExportMemory` in `src/web/app.js`) and by the Rust mirror `memory::export_full_memory` in `src/memory.rs` (used by `formal-ai memory export` as the new default). |
| R110 | Imports must accept both the legacy `demo_memory` document and the new `formal_ai_bundle` document so old exports continue to work. | Implemented by `FormalAiMemory.importFullMemory` (browser) and `memory::import_full_memory` (Rust). Both auto-detect the document header and fall back to the legacy `demo_memory` parser when no bundle header is present; covered by `import_full_memory_accepts_legacy_demo_memory` in `src/memory.rs`. |
| R111 | After import, surface known data migrations (e.g. seed-version mismatch) directly in the UI so the user can take action without reading code. | Implemented by `FormalAiMemory.suggestMigrations` (browser) and `memory::suggest_migrations` (Rust). `handleImportMemory` in `src/web/app.js` appends the resulting suggestions to the memory-status indicator; the CLI `memory import` / `bundle import` commands print each suggestion via `eprintln!("Migration: <message>")`. |
| R112 | The prefilled "Report issue" body must instruct users to wrap the `.lino` export in a `.zip` (GitHub does not yet accept `.lino` attachments), and must explicitly remind users to redact sensitive content before attaching. | Implemented in `createIssueReportBody` in `src/web/app.js`: the **Attach full memory (recommended)** block calls out the `.lino`/`.zip` limitation with per-OS zip instructions (macOS Compress, Windows Send to → Compressed folder, Linux `zip`) and a dedicated redaction reminder. |
| R113 | The CLI surface (`formal-ai memory export`, `formal-ai bundle export`) and the Rust library API must default to full-memory output as well, so every interface stays consistent. | Implemented by switching `formal-ai memory export` to write the full `formal_ai_bundle` by default in `src/main.rs`; `--events-only` opts back into the legacy `demo_memory` shape. `formal_ai::export_memory_full`, `import_memory_full`, `suggest_memory_migrations`, `BundleInfo`, and `ParsedBundle` are re-exported from the crate root in `src/lib.rs`. |
| R114 | Keep VISION.md, REQUIREMENTS.md, README.md, and the e2e/unit tests in lockstep with the new defaults so the documentation, the requirement matrix, and the regression coverage all describe the same behavior. | Implemented by this row plus the **Single-File Reproducibility** paragraph in `VISION.md`, the export/import notes in `README.md`, the rewritten `Export memory downloads a Links Notation file` Playwright e2e in `tests/e2e/tests/multilingual.spec.js` (now asserting `formal_ai_bundle` + `seed_files`), and the new `memory::tests::full_memory_round_trip_*` / `suggest_migrations_*` unit tests. |
