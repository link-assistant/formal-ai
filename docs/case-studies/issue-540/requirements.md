# Issue 540 Requirements

| ID | Requirement | Implementation |
| --- | --- | --- |
| R540-01 | Preserve the raw issue #540 material, related issue #494 material, prepared PR #645 state, and research. | `docs/case-studies/issue-540/raw-data/` and this case-study directory. |
| R540-02 | Dreaming must be available by default as background work in the core server and desktop, not only as a manual repair command. | `src/dreaming_runtime.rs` starts with `formal-ai serve`; `desktop/lib/dreaming.cjs` starts the Electron scheduler unless `FORMAL_AI_DESKTOP_DREAMING=off`. |
| R540-03 | Background dreaming must be low priority and yield to foreground work across platforms. | Core request guards and desktop idle probes defer dreaming while foreground work is active; desktop uses `nice -n 19` on Unix and `os.setPriority(...PRIORITY_LOW)` where supported. |
| R540-04 | Default dreaming may retain learning but must not free memory without consent. | The core runtime strips removal actions unless persisted auto-free-space consent exists; `formal-ai memory dream` remains plan-only unless explicitly applied. |
| R540-05 | Destructive application must keep the existing backup/confirmation safety boundary. | `run_memory` requires `require_destructive_confirmation` and can write a full-memory `--backup` before `apply_dreaming_plan`. |
| R540-06 | Reorganization must use recalculated frequency of use, not stale insertion order. | `src/dreaming.rs::usage_counts` scans current event text/evidence links; tests verify the more referenced duplicate is retained. |
| R540-07 | Duplicate cleanup must target recomputable/cache/intermediate records only. | `DreamingDurability` classification and duplicate keys are emitted only for `RecomputableCache` and `RecomputableIntermediate`. |
| R540-08 | Garbage collection must keep a free-space reserve, defaulting to 20% from real disk measurements and the actual incoming write size. | `src/storage_policy.rs` uses `fs2` filesystem statistics and passes the incoming byte count into `required_reclaim_bytes`. |
| R540-09 | The policy must preferentially forget refetchable/recomputable data and unused cached data. | Pressure candidates are restricted to deleted conversations, cache records, and intermediate records, sorted by low recalculated usage. |
| R540-10 | Raw events, learned experience, learning ledgers, and generalized algorithms must be retained. | `IrreplaceableRaw` and `RetainedLearning` are never reclaimable; tests assert raw and learning events are not selected under pressure. |
| R540-11 | If reclaimable data cannot satisfy the target, the system must report the need for bigger storage instead of over-deleting. | `DreamingPlan::requires_bigger_storage` is set when selected reclaimable bytes are insufficient; the CLI prints a migration warning. |
| R540-12 | The feature must be accessible to users and embedders. | Public re-exports in `src/lib.rs` and the `formal-ai memory dream` CLI surface. |
| R540-13 | The implementation must be documented and regression-tested. | `REQUIREMENTS.md`, `ARCHITECTURE.md`, this case study, `tests/unit/memory_maintenance.rs`, `tests/unit/docs_requirements_issue_540.rs`, and `desktop/scripts/dreaming.test.mjs`. |
| R540-14 | Dreaming must recalculate which topics the user interacts with most from memory links. | `src/dreaming/learning.rs::event_topic`/`learn_from_memory` produce ranked `TopicFrequency` records; tests assert the most-used topic ranks above chatter. |
| R540-15 | Dreaming must remember durable requirements without an English-only production cue list. | `requirement_statement` loads multilingual cues from `data/meta/dreaming-cues.lino` into `LearnedRequirement` records. |
| R540-16 | Dreaming must generalize each requirement into a meta-algorithm amendment baked into memory so new requirements are applied to similar future tasks. | `src/dreaming.rs::MetaAlgorithmAmendment` and `apply_dreaming_plan` materialize an idempotent, retained `meta_algorithm_amendment` event; verified by `dreaming_generalizes_requirements_into_meta_algorithm_amendments` and `applying_dreaming_plan_bakes_amendments_and_is_idempotent`. |
| R540-17 | Under pressure, dreaming must forget the specifics a retained amendment can reproduce first, keeping the generalization. | `DreamingActionKind::ForgetCoveredSpecific` and covered-specific-first pressure ordering; verified by `dreaming_forgets_covered_specifics_first_under_pressure`. |
| R540-18 | The dreaming meta-algorithm must be grounded, machine-readable data pinned to the live source. | `data/meta/dreaming-recipe.lino` and `tests/unit/specification/dreaming_meta_algorithm.rs`. |
| R540-19 | Stored amendments must be read and must change similar future task answers without the user repeating the rule. | `src/dreaming_application.rs` is integrated into both OpenAI-compatible protocol paths; `learned_amendment_changes_a_new_task_answer_without_repeating_requirement` proves the behavior. |
| R540-20 | A specific task may be considered covered only after replay verifies that the amendment reproduces its recorded output. | `replay_candidate_tasks` and `amendment_reproduces_specific` require exact normalized replay; unverified records are not selected by `ForgetCoveredSpecific`. |
| R540-21 | Dreaming must derive candidate tasks from frequent topics, simulate the amended algorithm, and expose pass/fail results. | `DreamingCandidateTask` records are produced by `src/dreaming/learning.rs` and rendered in the inspectable plan. |
| R540-22 | Failed simulation must preserve evidence for improving the meta-algorithm rather than claiming successful coverage. | Failed candidates remain in the plan with `passed=false`; the proposed retained amendment remains available for later learning while the specific record stays protected. |
| R540-23 | Dreaming must discover recurring task structures beyond explicit requirement phrases and retain the learning. | `mine_patterns` derives repeated structures from task data and `apply_dreaming_plan` stores `dreaming_pattern` records. |
| R540-24 | Automatic free-space cleanup must ask once, persist both acceptance and refusal, free only enough for the next operation, and prompt for larger storage when insufficient. | CLI and Electron consent flows use the `.auto-free-space` preference; `DreamingPlan::requires_bigger_storage` drives the migration prompt. |
| R540-25 | Recalculated usage must include cached and seed link records. | `classify_event` recognizes `seed_cache`/`seed_data`, while `usage_counts` follows ids through all searchable event fields and evidence links. |
| R540-26 | The machine-readable recipe must describe replay, pattern discovery, future-task application, real storage, and background consent. | The thirteen grounded steps in `data/meta/dreaming-recipe.lino` are checked against their live source files. |
| R540-27 | Documentation must use the repository's memory-links terminology consistently. | Issue #540 architecture, requirements, recipe, and case-study language use link terminology. |
| R540-28 | Formal AI must execute an issue-540 audit through Agent CLI and preserve the session plus every gap-to-generalization result. | `src/agentic_coding/dreaming_audit.rs` drives `write_file → run_command → final`; the generated `dreaming-gap-analysis.lino` and byte-for-byte pinned `agent-cli-session-dreaming-audit.json` are committed in this case study. |
| R540-29 | Live chat exchanges must land in the configured memory log and the full organic loop — record, dream, apply, changed future answer — must run through the production application path. | `record_chat_exchange` wired into the chat routes; `organically_recorded_chat_dreams_amendments_that_replay_through_production` and `chat_completions_route_records_live_exchange_into_configured_memory` prove the loop end-to-end. |
| R540-30 | Changing a topic's rule must revoke replay-verified coverage, and unverifiable records must fall back to normal eviction ordering instead of covered-specific forgetting. | `adding_a_requirement_revokes_coverage_and_preserves_the_stale_specific` and `failed_verification_falls_back_to_normal_eviction_ordering_under_pressure` in `tests/unit/memory_learning.rs`. |
| R540-31 | The auto-learning loop must consume failed replays: marker-grounded refinement resurrects lost rules and remaining failures persist as durable refinement records. | `refine_amendments_from_failures` accepts only projection marker lines; `failed_replay_refines_the_amendment_back_from_the_recorded_marker` and `failed_replays_are_preserved_as_refinement_records` verify resurrection and idempotent preservation. |
| R540-32 | Dreaming must synthesize genuinely new trials from mined patterns on top topics and gate task detection through the multilingual lexicon, not English-only kinds. | `dreaming_synthesizes_new_trials_from_numeric_patterns_on_top_topics` and `multilingual_task_kinds_are_replayed_as_candidates` against `data/meta/dreaming-lexicon.lino`. |
| R540-33 | The background runtime itself must be regression-tested: idle gating, mid-run cancellation, the `FORMAL_AI_DREAMING` opt-out, serve() wiring, locked atomic shared-log writes, and the desktop child's low OS priority. | `tests/unit/dreaming_runtime.rs` (e.g. `a_foreground_request_cancels_a_dreaming_run_mid_flight`, `write_locked_atomic_creates_parents_replaces_content_and_leaves_no_temp_files`) and the `PRIORITY_LOW` assertion in `desktop/scripts/dreaming.test.mjs`. |
