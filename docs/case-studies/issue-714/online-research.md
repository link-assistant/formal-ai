# Online research

Research was refreshed on 2026-07-15 from primary documentation.

- [OpenCode tools](https://opencode.ai/docs/tools/) documents separate `websearch`
  and `webfetch` tools. `websearch` is available with the OpenCode provider or
  when `OPENCODE_ENABLE_EXA` is truthy; this is why the wrapper enables the switch
  rather than inventing an embedded search response.
- [OpenCode permissions](https://opencode.ai/docs/permissions/) makes the client
  the approval boundary (`allow`, `ask`, or `deny`) for `bash`, `websearch`,
  `webfetch`, and other tools. Formal AI should emit the call and let this policy
  decide execution.
- [OpenAI function calling](https://developers.openai.com/api/docs/guides/function-calling)
  defines tool calls as model requests and tool outputs as values generated by the
  application. This supports retaining the returned client result while keeping
  execution outside Formal AI's API server.
- [Claude tool-use contract](https://platform.claude.com/docs/en/agents-and-tools/tool-use/how-tool-use-works)
  distinguishes client-executed tools from server-executed tools and requires the
  application to return `tool_result` blocks for client tools. The shared adapter
  therefore preserves call ids and results rather than flattening them into prose.
- [Gemini function calling](https://ai.google.dev/gemini-api/docs/function-calling)
  likewise states that executing function code is the application's
  responsibility and the result is sent back for a final response. This motivated
  full `functionCall`/`functionResponse` history translation.
- [`gh issue create`](https://cli.github.com/manual/gh_issue_create) documents the
  non-interactive title/body/repository flags used by the report recipe.

OpenCode Zen remains a model gateway, not the documented implementation of its
search tool. The integration therefore uses OpenCode's native search capability
when available and stays provider-independent at the Formal AI planner layer.
