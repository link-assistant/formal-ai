===== PR#637: feat(issue-558): complete human-gated auto-learning loop (R558-01ŌĆ”R558-06) via the agentic interface (merged=true)
## Issue #558 ŌĆö Auto Learning

Closes #558. This PR takes issue #558 from documentation-only to a **fully
implemented, human-gated, deterministic** auto-learning / self-healing loop,
wired end to end through the agentic interface with unit + integration
coverage. Every one of the issue's six requirements (`R558-01`ŌĆ”`R558-06`) is
now built, tested, and reachable through an external agent CLI (Codex /
OpenCode / Gemini / Agent CLI) or the in-repo driver.

### Design guardrails (preserved throughout)

- **Human-gated & proposal-only.** Nothing is ever auto-applied. Every artifact
  is a *record for review*, never an automatic write-back to source, seed, or
  the running UI.
- **Deterministic.** Every artifact is a pure function of its inputs ŌĆö no
  randomness, no wall-clock.
- **Neural inference is a NON-GOAL.** All reasoning is symbolic and links-based.
- **Byte-for-byte round-trips.** Every `source ŌåÆ links ŌåÆ source` projection
  reproduces the original bytes.
- **Cheap CI.** Exhaustive whole-repo proofs are `#[ignore]`d.

### What ships ŌĆö the eleven deterministic recipes

Each capability is a slice: library module + agentic recipe + planner routing +
example + unit test + integration test + docs. All eleven are reachable through
the agent-mode server, each keyed on disjoint keywords so the planner routes
unambiguously.

| # | Recipe | Requirement | Library |
|---|--------|-------------|---------|
| Closed self-healing loop | `self_heal` | R558-01/03 | `src/self_healing.rs` |
| Self-AST projection | `self_ast` | R558-04 | `src/self_source_graph.rs` |
| Whole-repo source graph | `source_graph` | R558-04/05 | `src/self_source_graph.rs` |
| Learning ledger (promotion protocol) | `ledger` | R558-03 | `src/learning_ledger.rs` |

| **Rebuild & reattach** | **`rebuild`** | **R558-06** | **`src/rebuild_plan.rs`** |
| ŌĆ”plus the meaning/formalization/diagram recipes |  |  |  |

### The final requirement closed by this session ŌĆö R558-06

*"Formal AI must be able to recompile and reattach the improved code to the UI."*

- `src/rebuild_plan.rs` composes a deterministic, human-gated `RebuildPlan`
  that is **derived from an already-accepted change** (`for_accepted_change`
  takes an `AcceptedChange`, which only exists after a green benchmark gate
  *and* an explicit human approval), so a rebuild can never precede acceptance.
- It **grounds every reattached UI artifact** against the real repository bytes
  and the owned manifest ŌĆö `Cargo.toml`, the server/CLI entry (`src/main.rs`,
  resolved through `owned_manifest`), the worker glue
  (`src/web/formal_ai_worker.js`), and the UI entry (`src/web/index.html`). The
  regenerated `formal_ai_worker.wasm` is deliberately the pipeline's *output*,
  referenced by the steps rather than pinned as an input.
- The plan is an ordered five-step pipeline (recompile ŌåÆ regenerate worker ŌåÆ
  reattach ŌåÆ hot-swap ŌåÆ verify), and **every step is observable** (what proves
  it happened) **and reversible** (how to roll it back), so keeping the swap
  stays a reviewable human decision.
- Nothing is rebuilt or restarted: the plan is the reviewable product. Reachable
  as the eleventh recipe (`src/agentic_coding/rebuild_plan.rs`), emitting the
  plan *live* as `rebuild-and-reattach.lino` (keyed on `reattach`, disjoint from
  the source-graph recipe's `recompile`).

### Requirements matrix ŌĆö all six implemented

- **R558-01** Failure never simply fails ŌåÆ `src/self_healing.rs` `RepairCase`.
- **R558-02** Rethink the meta algorithm using itself ŌåÆ `src/repair_strategy.rs`

- **R558-06** Recompile & reattach to the UI ŌåÆ `src/rebuild_plan.rs` (this session).

===== PR#601: Make our meanings and words more detailed (grammatical number, part of speech, Wikidata grounding) (merged=true)
## Issue

Closes #538 ŌĆö "Make our meanings and words more detailed".

The issue starts from a concrete, verifiable observation: the tomato meaning lists the Russian surfaces `ą┐ąŠą╝ąĖą┤ąŠčĆ`, `ą┐ąŠą╝ąĖą┤ąŠčĆčŗ`, `čéąŠą╝ą░čé` **without saying which is singular or plural**, and `ą┐ąŠą╝ąĖą┤ąŠčĆ` has a plural while its synonym `čéąŠą╝ą░čé` does not. Its central *method* requirement is that the task be solved **by driving Formal AI through its own [Agent CLI](https://github.com/link-assistant/agent)** ŌĆö *"you don't read or edit code or files yourself, you only use Agent CLI with Formal AI server connected to do it"* ŌĆö never by hand-editing and never by deferring parts to a roadmap.

This PR does exactly that. See [`docs/case-studies/issue-538/`](docs/case-studies/issue-538/README.md) for the full analysis.

## How the change is produced: the Agent CLI + Formal AI, not a human editor

The change is authored by an **in-repo agentic driver** (`src/agentic_coding/`) that plays the Agent CLI against the OpenAI-compatible Formal AI server (`formal-ai serve`), offline and deterministically, through a real tool loop:

```
web_search ŌåÆ web_fetch ŌåÆ write_file ŌåÆ run_command (cat verify) ŌåÆ final
```

The driver's `write_file` step emits the enriched meaning block, and the committed seed data is **byte-for-byte identical** to what the driver writes ŌĆö `tests/unit/issue_538_agentic.rs` asserts `seed == driver-output`, so the Formal-AI-driven recipe is the author, not a human, and it cannot silently regress into hand-editing. The exact sessions that produced the change are committed as [`agent-cli-session.json`](docs/case-studies/issue-538/agent-cli-session.json) (tomato), [`agent-cli-session-potato.json`](docs/case-studies/issue-538/agent-cli-session-potato.json) (potato), [`agent-cli-session-diagram.json`](docs/case-studies/issue-538/agent-cli-session-diagram.json) (recipe diagrams), and [`agent-cli-session-self-ast.json`](docs/case-studies/issue-538/agent-cli-session-self-ast.json) (self-AST census) ŌĆö the *"json file with Agent CLI session that solved this exact task"* the issue asks for.

**Generality (not hardcoded):** the recipe is a **concept registry** and is driven for **four artifacts with four differently-worded requests** ŌĆö tomato meaning, potato meaning, and two non-lexeme axes (the generated recipe diagrams and the self-inspection CST/AST census) ŌĆö routed by the planner from each request's own wording. Tests assert the differently-worded meaning requests reach the two distinct concepts and that the third and fourth requests drive entirely different artifacts, so a passing run is evidence the recipe generalises rather than pattern-matching one answer.

**When the tool couldn't do it, we extended the tool:** the tomato-only recipe was refactored into the concept registry, a real TOCTOU race in `AgentWorkspace::for_prompt` was fixed, and two further recipes were added so the same Agent CLI generates architecture diagrams and stores its own CST/AST ŌĆö rather than hand-finishing and deferring.

**No hardcoded answers ŌĆö every surface is *derived* from real data:** `src/agentic_coding/meaning_detail.rs` holds no hand-typed answer strings. Each enriched surface is derived from real, checked-in Wikidata lexeme JSON (`data/cache/wikidata/lexeme/{L7993,L3526,L170542,L3784}.json`, embedded via `include_str!`) parsed by a general `serde_json` algorithm. The selection is **lemma-anchored and case-key-matched**: the singular form is the one whose text equals the lexeme's lemma, and the plural is the form whose non-number grammatical features match the singular's ŌĆö general for any case paradigm and referencing **no hardcoded case id**. Hardcoded strings live only in tests; the four seed blocks are reproduced byte-for-byte from the source JSON, so correctness is proven against real Wikidata data rather than a memorised answer.

===== PR#638: Generate, classify, and answer questions via the Agent CLI (issue #527) (merged=true)
## Summary

Closes #527. This PR delivers the issue #527 vision ŌĆö *generate all possible
questions from smallest to largest and answer them* ŌĆö and, per the maintainer's
linchpin instruction, **executes the task by driving Formal AI through its own
Agent CLI** rather than hand-editing files, wiring the answers into auto-learning
through the meta algorithm's existing recipe pattern.

There are two layers:

1. **A lazy, configurable question-generation core** (`src/question_generation.rs`)
   ŌĆö an infinite iterator of candidate questions ordered smallest-first, ranked
   by frequency evidence, classified by grammar and logical meaning, with a
   configurable "what counts as a question" surface, and an answer stream that
   delegates to the existing `FormalAiEngine`.
2. **The eleventh agentic recipe** (`src/agentic_coding/question_catalog.rs`) that
   drives the Agent CLI to enumerate, classify, answer, and record the catalog as
   reviewable Links Notation, with a recall table that feeds auto-learning without
   ever silently changing solver behaviour.

## What it does

- **Lazy smallest-to-largest stream.** `QuestionGenerator` implements
  `Iterator<Item = GeneratedQuestion>`, advancing by word count (1 word, 2 words,
  ŌĆ”) without materializing the infinite space.
- **Frequency-tiered vocabulary averaged across corpuses.**
  `QuestionWord::from_corpus_scores` records averaged multi-corpus frequency;
  `QuestionGenerationConfig` defaults to the issue's tier policy (top 10% at
  one/two words, 5% at three, 2.5% at four, halving thereafter with a practical
  floor). Ranking and grammar roles are **grounded in the committed seed lexicon**
  `data/seed/question-generation-lexicon.lino`, not hardcoded in production code.
- **Two classification gates.** Every candidate carries
  `QuestionGrammarClass::{Fragment, Grammatical, Ungrammatical}` and, among the
  grammatical ones, `LogicalMeaningClass::{Meaningful, OpenSlot, NotMeaningful}`,
  plus a combined `GeneratedQuestionClass`.
- **Configurable acceptance.** `QuestionAcceptance` selects any question-like,
  grammatical, or grammatical-and-meaningful candidates ŌĆö the issue's "what counts
  as a question" definition is a config surface, not a hardcoded rule.
- **Best-possible answers via the existing engine.**
  `generated_question_answers(config)` composes the generator with
  `FormalAiEngine::answer`, so answering stays on the universal solver path.
- **Driven by the Agent CLI.** The `question_catalog` recipe routes a
  differently-worded request through the deterministic planner

===== PR#618: Solve the whole class of external-entity questions by reasoning (#571) (merged=true)
## Summary

Issue #571 asks for the ability to solve an **entire class** of similar
externally verifiable questions using reasoning and logic in the meta algorithm
ŌĆö not just the specific Russian subscription-pricing example. The maintainer
explicitly flagged that a seed-vocabulary-only fix was too narrow.

This PR keeps the seed-vocabulary path for the exact reported prompt **and**
adds a genuine reasoning rule that generalizes to the whole class by reasoning
about the *referent* of a question rather than memorised topic words.

## The reasoning rule

`extract_externally_verifiable_question` (in
`src/solver_handlers/web_search_intent.rs`) routes a prompt to the
source-gathering web-research plan when three structural conditions all hold:

1. **Interrogative** ŌĆö the prompt opens with a seeded question opener (any
   language) or ends with a question mark (`?` / fullwidth `’╝¤`).
2. **Names a referential external entity** ŌĆö it contains a Latin token written
   with *interior capitalisation* (`ChatGPT`, `OpenAI`, `GitHub`, `iPhone`,
   `TypeScript`), the orthographic signature of an engineered brand/product
   name whose current facts (pricing, availability, features, history) live on
   the public web rather than in seed memory.
3. **Not locally resolvable** ŌĆö it is neither a seeded concept lookup nor a
   self-introduction / capability / non-referential-subject question, so those
   handlers keep their prompts.

Because interior capitalisation is a property of the Latin brand token itself,
the rule fires identically for English, Russian, Hindi, and Chinese prompts and
across any topic. All-caps acronyms (`BSD`, `ML`, `USD`), plain capitalised
proper nouns (`Claude`, `Tesla`, `Wikipedia`), and Title-Cased phrases
(`Hive Mind`) deliberately do **not** match, so concept-lookup, coding, and
unknown-reasoning prompts are untouched.

## Why this satisfies "entire class, not just the example"

The rule is a pure structural test with **no per-product and no per-language
word list to maintain**. The Russian annual-discount prompt for Claude Max and
ChatGPT Pro is handled as one instance of the general class, alongside
unrelated, non-commercial questions such as "Is the iPhone 16 waterproof?" or
"ąÜąŠą│ą┤ą░ OpenAI ą▓čŗą┐čāčüčéąĖą╗ą░ čüą▓ąŠčÄ ą┐ąĄčĆą▓čāčÄ ą╝ąŠą┤ąĄą╗čī?".

===== PR#564: Support repository resource summarization (files and folders) (merged=true)
## Summary

Closes #563. Generalizes repository summarization from **files only** to **any repository resource ŌĆö files *and* folders** ŌĆö by following the meta algorithm's *describe ŌåÆ formalize ŌåÆ decompose ŌåÆ solve ŌåÆ compose* loop, applied recursively over the repository tree.

The meta algorithm must solve any task by description, formalization to the meta language, decomposition, testing, and composition ŌĆö so summarization now treats a repository as a recursive tree of resources rather than a flat list of files.

## What changed

- **`src/summarization/resource.rs` (new):** the generalization layer.
  - `RepositoryEntry` ŌĆö a filesystem-free input tree (`File { path, content }` / `Directory { path, children }`), so the layer stays deterministic and testable; a real directory walk builds the tree and hands it in.
  - `formalize_repository_resource` dispatches on kind: files reuse `formalize_repository_file` unchanged; directories recurse through `formalize_repository_directory`, aggregating recursive file/subdirectory counts and total lines/bytes.
  - A folder is summarized by the **decompose ŌåÆ summarize ŌåÆ compose** loop: split the directory into children, summarize each child on its own (files via `file.rs`, subdirectories by recursion), then compose behind an aggregate identity sentence.
  - **Recursion depth is bounded by the summarization mode ladder** (`SummarizationMode::one_step_shorter`), not an ad hoc depth cap: a `Full` folder describes its children in `Standard`, theirs in `Short`, deeper as `Topic`.
  - `summarize_repository_resource` is the single general entry point and reproduces `summarize_repository_file` exactly for file inputs (the file behavior is subsumed, not replaced).
  - `RepositoryDirectoryFormalization::links_notation` renders link-native `repository_directory` evidence.
- **`src/summarization/file.rs`:** exposes `RepositoryFileFormalization::summary` so directory recursion reuses file summaries.
- **Exports** wired through `mod.rs`, `src/lib.rs`, and the `tests/source` mirror.

===== PR#524: fix(ci): resolve 'No space left on device' in Deploy Demo job (issue #523) (merged=true)
## Summary

Fixes the only hard failure in CI/CD run [27703726124](https://github.com/link-assistant/formal-ai/actions/runs/27703726124): the **Deploy Demo to GitHub Pages** job crashed with `System.IO.IOException: No space left on device`.

### Root cause
The `deploy-demo` job restored a **shared, multi-gigabyte `target/` cache** keyed `${{ runner.os }}-cargo-${{ hashFiles('**/Cargo.lock') }}` ŌĆö the *same* key the `lint` job (`cargo clippy --all-targets --all-features`) and `test` job populate. On top of that restored tree the job built the web vendor bundle, ran `cargo doc`, duplicated `target/doc` into the Pages artifact, and tarred it ŌĆö exhausting the `ubuntu-latest` runner's ~14 GB root disk. The runner worker process itself then crashed while writing its diagnostic log.

This was a **self-introduced regression**: PR #479 added the `cargo doc` step + `target` cache to `deploy-demo`. The upstream [rust pipeline template](https://github.com/link-foundation/rust-ai-driven-development-pipeline-template)'s doc-deploy job caches **no** `target/` and is unaffected ŌĆö so there is **no upstream bug to file** (full template comparison in the case study).

### Fix (`.github/workflows/release.yml`, `deploy-demo` job)
- **Cache only `~/.cargo/registry` + `~/.cargo/git`** under a dedicated `*-cargo-docs-*` key ŌĆö the bloated `target/` tree is never restored here. `cargo doc --no-deps --lib` rebuilds its own small `target/` from the cached registry.
- **Free unused pre-installed SDKs** (dotnet / android / ghc / codeql) and prune docker images before the heavy build (~20ŌĆō30 GB reclaimed).
- **Log `df -h`** before/after cleanup so disk pressure is visible on every run (verbose/debug for future iterations).

### Other CI signals (triaged, no code change)
- ŌÜĀ’ĖÅ *Node.js 20 deprecated ŌĆ” `actions/github-script@60a0d83ŌĆ”`* ŌĆö transitive inside `codecov/codecov-action@v5`; not referenced by our workflows. External, non-blocking.
- ŌÜĀ’ĖÅ *Rust file approaching 1000-line limit* (├Ś10) ŌĆö intended advisory from the repo's own file-size lint, not a false positive.

### Case study
Full evidence + analysis (timeline, requirements, root causes, solutions, template comparison) in [`docs/case-studies/issue-523/`](https://github.com/link-assistant/formal-ai/tree/issue-523-d811f560cba2/docs/case-studies/issue-523).

## Verification
- `release.yml` parses cleanly (`yaml.safe_load`).
- Deploy cache no longer lists `target` and uses an isolated key ŌåÆ cannot restore the multi-GB lint/test tree.
- Final proof: the next green **Deploy Demo to GitHub Pages** run on `main`.
