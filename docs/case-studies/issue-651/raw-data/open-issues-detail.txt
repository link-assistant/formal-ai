=========== ISSUE 650 ===========
TITLE: formal-ai with: fix codex 'hi' misroute (/responses merges instructions), uniform interactive/non-interactive mode for all tools, summarization handling, --globally alias
STATE: open
LABELS: 
BODY:
Follow-up to #647 (shipped in 0.278.0). The `formal-ai with` wrapper now auto-starts a temporary `--agent-mode` server, keeps config temporary, and supports all 8 tools — verified working. But making it behave **consistently and correctly across every tool** exposes four related defects. Filing them together because they share one root goal: `formal-ai with <tool>` should Just Work — interactive by default, one-shot on demand, `hi` answered as `hi`, no accidental summarization, and never touching global config.

All evidence below is from **`formal-ai 0.278.0`** driving the locally-installed CLIs (codex `v0.144.1`, gemini, qwen, `@link-assistant/agent`, opencode). `claude` was intentionally **not** run to avoid interfering with a running Claude process; its behavior is inferred from the shared adapters.

---

## 1. `hi` is misrouted for tools that use the OpenAI **Responses** API (codex)

`formal-ai with codex "hi"` returns a long "fetch-JSON / compute-mean-median" Rust program instead of a greeting. Root cause is proven by request tracing (`FORMAL_AI_TRACE_REQUESTS=1`):

- **agent / opencode / gemini / qwen** send `POST /api/openai/v1/chat/completions`; the system prompt is a separate `system`-role message and the last user message is a clean `"hi"`. → server replies **"Hi, how may I help you?"** ✅
- **codex** sends `POST /api/openai/v1/responses` with a 20,751-char `instructions` field + a `user` item whose content is `[{"type":"input_text","text":"hi"}]`. → server **misroutes**. ❌

The `/responses` handler **merges `instructions` into the user query** instead of treating it as separate steering context. Minimal repro against the server directly:

```sh
# A) /responses, bare "hi"  → CORRECT
curl -s localhost:8080/api/openai/v1/responses -H 'content-type: application/json' \
  -d '{"model":"formal-ai","input":[{"role":"user","content":[{"type":"input_text","text":"hi"}]}]}'
# → "Hi, how may I help you?"

# B) /responses, "hi" + an instructions field  → WRONG
curl -s localhost:8080/api/openai/v1/responses -H 'content-type: application/json' \
  -d '{"model":"formal-ai","instructions":"You are a coding agent... Decompose requests into sub-tasks.","input":[{"role":"user","content":[{"type":"input_text","text":"hi"}]}]}'
# → "I could not determine `You are a coding agent... \nhi` ..."  (instructions concatenated onto the query)

# C) /chat/completions, "hi" + the SAME prompt as a system message  → CORRECT
curl -s localhost:8080/api/openai/v1/chat/completions -H 'content-type: application/json' \
  -d '{"model":"formal-ai","messages":[{"role":"system","content":"You are a coding agent... Decompose requests into sub-tasks."},{"role":"user","content":"hi"}]}'
# → "Hi, how may I help you?"
```

**Fix:** the `/responses` handler must isolate the final user turn and treat `instructions` (and `developer`-role items) as system/steering context — exactly as `/chat/completions` already does — so `formal-ai with codex "hi"` answers `hi`.

**Acceptance:** `formal-ai with codex "hi"` → a greeting, not a program. Repro B returns the same as A.

---

## 2. No-message runs should enter **interactive** mode — behavior is inconsistent per tool

`formal-ai with <tool>` with no prompt behaves differently for every tool (tested with stdin closed):

| Tool | No-message result |
| --- | --- |
| **opencode** | ✅ launches its interactive TUI ("Ask anything…") |
| **agent** | ❌ exits 0, does nothing |
| **gemini** | ❌ `Error: gemini exited with status 42` — "No input provided via stdin" |
| **qwen** | ❌ `Error: qwen exited with status 1` |
| **codex** | ❌ `Error: codex exited with status 1` — "No prompt provided via stdin" |

Every underlying tool actually **defaults to interactive** when run directly (codex forwards to its interactive CLI; gemini/qwen "default to interactive, use `-p` for headless"; agent runs "interactive or stdin mode"). The wrapper breaks that because it does not allocate a TTY / forward interactive intent, so the tools hit "no stdin" and error.

**Requested behavior (uniform across all tools):**
- **No message → interactive mode by default** (allocate a PTY / pass the tool's interactive invocation).
- **Explicit control that works for every tool**, regardless of whether a message is given:
  - `--interactive` → force interactive (even with a message: e.g. gemini/qwen `-i/--prompt-interactive`, agent `--interactive`).
  - `--non-interactive` (a.k.a. `--print`/one-shot) → force headless (e.g. gemini/qwen `-p`, codex `exec`, agent `-p`).
- These wrapper flags map to each tool's native equivalent from seed data, so the UX is identical across tools.

**Acceptance:** `formal-ai with <tool>` (no args) drops into that tool's interactive prompt for **all** of codex/opencode/agent/gemini/qwen/claude/grok/aider; `--interactive` / `--non-interactive` force the mode for each.

---

## 3. Disable summarization by default for **all** tools; if a tool can't disable it, the server must handle summarize requests

#647 disabled summarization for the `agent` CLI (via `--no-summarize-session`). But **codex, gemini, qwen expose no user-facing no-summarize / no-compaction flag**, so the wrapper cannot turn it off for them. And formal-ai has effectively unbounded context, so any summarization those tools trigger is pure waste — **and it currently fails**:

```sh
curl -s localhost:8080/api/openai/v1/chat/completions -H 'content-type: application/json' \
  -d '{"model":"formal-ai","messages":[{"role":"user","content":"Summarize the following conversation: User asked about the weather. Assistant said it is sunny. User thanked the assistant."}]}'
# → "I could not determine `the weather. Assistant said it is sunny...` ... cannot infer a verified answer."
```

**Requested behavior:**
- For every tool that **has** a disable flag, the wrapper passes it by default (with `--summarize` to opt back in — already exists).
- For every tool that **has no** disable flag, formal-ai the **server** must correctly satisfy a summarization request — whatever prompt shape that tool sends for summarize/compact — and return a valid summary instead of "I could not determine…".

**Acceptance:** driving any tool through `with` never emits a failed-summarization error; an explicit "summarize this conversation" request to the server returns a real summary.

---

## 4. `--globally` spelling; keep confirming temp-config-only

Two smaller items:

- The permanent-config flag is `-g` / `--global`; **`--globally` is rejected** (`error: unexpected argument '--globally'`). Requested: accept `--globally` (as an alias at minimum) since that's the natural spelling.
- **Temp-config-only is currently holding** — verified: checksums of all global configs (`~/.codex/config.toml`, `~/.config/opencode/*`, `~/.config/link-assistant-agent/*`, `~/.gemini/settings.json`, `~/.qwen/settings.json`) were **byte-identical** before/after every non-global run in this test matrix (interactive attempts, one-shot, no-message, codex, agent). codex confirmed it by using a temporary `codex_home` under `/var/folders/...`. Please keep this invariant covered by tests as the interactive/PTY work in item 2 lands (a PTY path must not start writing real config).

---

## Consolidated acceptance criteria

- [ ] `formal-ai with codex "hi"` returns a greeting (Responses `instructions` no longer merged into the query; `/responses` isolates the last user turn like `/chat/completions`).
- [ ] `formal-ai with <tool>` with no message enters interactive mode for **every** supported tool.
- [ ] `--interactive` / `--non-interactive` force the mode uniformly across all tools (mapped to each tool's native flag via seed data).
- [ ] Summarization/compaction disabled by default where the tool supports it; where it doesn't, the server returns a valid summary for summarize/compact requests instead of failing.
- [ ] `--globally` is accepted (alias of `-g/--global`).
- [ ] Non-global runs never write persistent config for any tool (regression test, incl. the new interactive/PTY path).
