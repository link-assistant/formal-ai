## Summary

Add a documented guide â€” e.g. `docs/testing/agentic-cli-tools.md` â€” describing how to test external agentic CLIs (codex, opencode, gemini, and our own `agent`) against a local Formal AI server. This session produced a repeatable methodology (fixtures, a logging proxy for provenance, phrasing matrices, marker-based assertions) that found a series of real bugs (#620, #621, #622, #624, #626, #627). Capturing it as a guide preserves that know-how and gives contributors a runbook for reproducing/extending the checks. It also seeds the concrete e2e suite proposed in #625.

## Why

Agent-mode behavior is easy to get wrong in ways plain unit tests miss: a request returns HTTP 200 but the model never emits the tool-call, or emits it in a schema the client can't parse, or a filename is misread as a URL. The only reliable way to catch these is to drive the real CLIs end-to-end and assert on observed behavior â€” with a way to *prove* the answer came from our server. That workflow should be written down.

## Proposed guide outline

### 1. Setup
- Build wrapper + server: `cargo build --bin with-formal-ai --bin formal-ai`.
- Install/update the client CLIs the same way they ship (bun global): `bun add --global @openai/codex@latest opencode-ai@latest @google/gemini-cli@latest @link-assistant/agent@latest`. Note pinning versions for reproducible CI.
- Start the server in agent mode: `formal-ai serve --agent-mode --host 127.0.0.1 --port 8080`.

### 2. Per-CLI invocation cheatsheet (what "just works" vs. what needs flags)
- **opencode**: `with-formal-ai opencode run "<prompt>"` (clean).
- **agent** (ours): OpenAI-compatible; configure via env only with `LINK_ASSISTANT_AGENT_CONFIG_CONTENT` (inline OpenCode-shaped provider JSON) + `FORMAL_AI_API_KEY`; run `agent --model formalai/formal-ai --permission-mode auto -p "<prompt>"`. (Wrapper support tracked in #621.)
- **codex**: uses the Responses wire API; needs `--skip-git-repo-check` to run outside a git repo; known tool-arg mismatch (#626).
- **gemini**: needs `GEMINI_DEFAULT_AUTH_TYPE=gemini-api-key`, `GEMINI_CLI_TRUST_WORKSPACE=true`, and isolation from `~/.gemini` cached OAuth (#620); headless `-p` sends no tool declarations.
- Redirect stdin from `/dev/null` for non-interactive runs; strip ANSI when scraping output.

### 3. Provenance via a logging proxy
- Put a logging reverse proxy between the CLI and the server and point the CLI at it with `with-formal-ai --base-url http://127.0.0.1:<proxyport>`.
- Assert every model call carried `model=formal-ai` (so no client-side model answered), and log per request: path, tools offered, tool-calls returned (name + arguments), status.
- Cover all three protocol paths: OpenAI `chat/completions` (opencode, agent), OpenAI `responses` (codex), Gemini `streamGenerateContent` (gemini) â€” including streamed SSE bodies. (A native Rust proxy is proposed in #625; until then a throwaway script works but shouldn't live in the repo.)

### 4. Fixtures and marker-based assertions
- Create a fixture workspace with known files and **distinctive content markers**, e.g. `alpha.txt` â†’ `ALPHA_MARKER_11111`, `beta.md` â†’ `BETA_MARKER_22222`, `gamma.json` â†’ `{"gamma_marker":"GAMMA_33333"}`, `subdir/nested.log` â†’ `NESTED_MARKER_44444`.
- A test passes only if the expected marker (or real filename, for listing) appears in the CLI's final answer â€” not merely exit code 0. Exit 0 means "the CLI ran," not "the task succeeded."

### 5. Phrasing matrices (test many wordings, expect natural conversation)
For each capability, run direct + natural-language + multi-step variations, because routing is wording-sensitive:
- **List files**: "run ls", "list the files in the current directory" (work today) vs. "what files are in this folder?", "show me the contents of this directory" (fail today â€” #624).
- **Read a file**: "read the file X", "cat X", "show me the contents of X", "what does X say?", "open X and tell me what's inside", "print the first line of X", "what is the value of Y in X?", plus multi-step "list the files then read the first one" (all failing today â€” #627).
- Include **negative cases** (prompts that should NOT trigger a tool call) to catch over-triggering.

### 6. Reading the results
- Classify each run: content/marker present? fell back to "could not determine" / "requires Agent mode" / URL-misparse? tool-call emitted in the proxy log?
- Compile failures into an issue with the full matrix and a minimal `curl` server-side repro (drop the CLI, send the same `messages` + `tools` array directly) so maintainers can reproduce without the CLI.

### 7. Wire it into CI
- Turn the matrices into an e2e job that runs on every PR (see #625): start server + proxy, install pinned CLIs, run matrices, assert on proxy-log + CLI output. Fail the build on regressions in intent-routing or tool-calling.

## Deliverable
A `docs/testing/agentic-cli-tools.md` (or under `docs/ci-cd/`) capturing the above, with copy-pasteable commands and the fixture/marker convention, cross-referenced from CONTRIBUTING. Should link the living examples: #624, #626, #627 (found with this method) and #625 (the CI suite it feeds).

TITLE: Add a built-in Rust logging reverse proxy (formal-ai proxy) for provenance/routing testing
CLOSED: 2026-07-04T00:18:01Z
BODY:
## Summary

Add a first-class **logging reverse proxy** to formal-ai (e.g. `formal-ai proxy --listen 127.0.0.1:8090 --upstream http://127.0.0.1:8080 --log proxy.jsonl`) for observing clientâ†”server traffic during testing. It sits in front of `formal-ai serve` and records, per request: path, `model`, the `tools` offered, and the response's `tool_calls` / content â€” so you can *prove* which server answered and see exactly how each request was routed.

## Motivation

While testing `with-formal-ai opencode` tool-calling, I needed to verify two things that are otherwise invisible:
1. **Provenance** â€” that answers came from the Formal AI server and not from any model on the client side.
2. **Routing** â€” whether the server returned a `bash` tool-call or a fallback text answer for each phrasing.

I threw together a Python proxy to do this, but (a) it doesn't belong in this Rust codebase, and (b) it couldn't cleanly parse streamed (SSE) responses. A native Rust proxy shipped with formal-ai would make this kind of verification reproducible in CI and for issue repros, and would handle streaming since it can pass bytes through untouched while still summarizing.

Concretely it would have made the routing-gap issue (natural-language file-listing requests not mapping to a tool-call) trivial to demonstrate and regression-test.

## Proposed behavior

- `formal-ai proxy --listen <addr> --upstream <url> [--log <path>] [--body]`
- Transparently forwards every request/response (including streaming) to `--upstream`.
- Appends one JSON line per exchange: method, path, request `model`, request `tools` (names), response `model`, response `tool_calls` (name + arguments), a content preview, and status.
- `--body` optionally logs full request/response bodies for deep debugging.

## Example implementation (std-only, blocking, no new deps)

A minimal starting point that forwards bytes and logs a summary. It reads the upstream response fully for non-streamed bodies; for streamed responses it can stream through and skip the JSON summary (or be extended to parse SSE `data:` lines). This is deliberately dependency-free so it can live behind a small subcommand; a production version would likely reuse the crate's existing async HTTP stack.

```rust
use std::io::{BufRead, BufReader, Read, Write};
use std::net::{TcpListener, TcpStream};
use std::sync::mpsc::Sender;

/// One request/response summary line appended to the log.
#[derive(Debug, Default)]
struct Exchange {
    method: String,
    path: String,
    req_model: Option<String>,
    req_tools: Vec<String>,
    status: u16,
    resp_model: Option<String>,
    resp_tool_calls: Vec<(String, String)>, // (name, arguments)
    resp_content_head: String,
}

fn main() -> std::io::Result<()> {
    // formal-ai proxy --listen 127.0.0.1:8090 --upstream 127.0.0.1:8080 --log proxy.jsonl
    let listen = "127.0.0.1:8090";
    let upstream = "127.0.0.1:8080";
    let log_path = "proxy.jsonl";

    let (tx, rx) = std::sync::mpsc::channel::<String>();
    // dedicated writer thread keeps log lines ordered and non-blocking for handlers
    std::thread::spawn(move || {
        let mut f = std::fs::File::create(log_path).unwrap();
        for line in rx {
            let _ = writeln!(f, "{line}");
        }
    });

    let server = TcpListener::bind(listen)?;
    eprintln!("formal-ai proxy on {listen} -> {upstream}, logging to {log_path}");
    for client in server.incoming() {
        let client = client?;
        let upstream = upstream.to_string();
        let tx = tx.clone();
        std::thread::spawn(move || {
            if let Err(e) = handle(client, &upstream, tx) {
                eprintln!("proxy error: {e}");
            }
        });
    }
    Ok(())
}

fn handle(client: TcpStream, upstream: &str, tx: Sender<String>) -> std::io::Result<()> {
    let mut reader = BufReader::new(client.try_clone()?);
    let mut client_out = client;

    // --- read request line + headers ---
    let mut request_line = String::new();
    reader.read_line(&mut request_line)?;
    let mut parts = request_line.split_whitespace();
    let method = parts.next().unwrap_or("").to_string();
    let path = parts.next().unwrap_or("").to_string();

    let mut headers = Vec::new();
    let mut content_length = 0usize;
    loop {
        let mut line = String::new();
        reader.read_line(&mut line)?;
        if line == "\r\n" || line.is_empty() {
            break;
        }
        if let Some(v) = line.to_ascii_lowercase().strip_prefix("content-length:") {
            content_length = v.trim().parse().unwrap_or(0);
        }
        headers.push(line);

TITLE: Expose reasoning/thinking in standard fields (reasoning_content, Anthropic thinking blocks) + stream it, so thinking-capable CLIs can display it
CLOSED: 2026-07-03T07:12:19Z
BODY:
## Summary

Formal AI produces a rich, inspectable reasoning trace (`thinking_steps`) for every answer â€” but it is exposed only in a **non-standard** field, and not streamed. As a result, agent CLIs that can **display model thinking** (codex reasoning summaries, Claude/Anthropic `thinking` blocks, DeepSeek-style `reasoning_content`) show **no reasoning** when driving Formal AI, even though the trace exists. Our reasoning transparency â€” a core selling point per `VISION.md` â€” is invisible to every external client.

Our APIs should expose thinking in each protocol's **standard reasoning field**, and stream it, so any thinking-capable CLI can render it.

## Evidence

### OpenAI Chat Completions â€” thinking is in a custom field, not `reasoning`
```
$ curl -s .../v1/chat/completions -d '{"model":"formal-ai","messages":[{"role":"user","content":"hi"}]}'
message fields: ['role', 'content', 'thinking_steps']
has reasoning: False | has reasoning_content: False | has thinking_steps: True
```
The trace lives in `message.thinking_steps` (Formal-AI-specific). OpenAI-compatible clients look for `message.reasoning` / `message.reasoning_content` (the de-facto field used by DeepSeek-R1, OpenRouter, vLLM, etc.), so they display nothing.

### Anthropic Messages â€” no `thinking` content block
```
$ curl -s .../v1/messages -d '{"model":"formal-ai","max_tokens":100,"messages":[{"role":"user","content":"hi"}]}'
content block types: ['text']
has thinking block: False
```
Anthropic clients render reasoning from `thinking` content blocks (`type: "thinking"`). Our `/v1/messages` emits only a `text` block, so Claude-style CLIs show no reasoning.

### Streaming â€” thinking is never streamed
Because `/v1/chat/completions` streaming is malformed (#604) and `/v1/responses` doesn't stream (#602), even the custom `thinking_steps` never arrives as incremental reasoning deltas. Thinking-capable clients render reasoning **live** from streamed deltas; a single final blob (or a non-standard field) can't drive that UI.

## Proposed behavior

Expose the existing `thinking_steps` trace through each protocol's standard reasoning channel, in both non-streaming and streaming responses:

- **OpenAI Chat Completions**: populate `message.reasoning_content` (and/or `message.reasoning`) with the rendered thinking, in addition to keeping `thinking_steps` for richness. In streaming, emit reasoning as `choices[].delta.reasoning_content` deltas before the content deltas.
- **OpenAI Responses**: emit `reasoning` output items / `response.reasoning_summary_text.delta` events in the stream (what codex's reasoning-summary UI consumes).
- **Anthropic Messages**: emit a `thinking` content block (and `content_block_delta` of `thinking_delta` when streaming) ahead of the `text` block.
- **Gemini** (per #603 S4): map thinking onto Gemini's `thought`-part convention.

Keep `thinking_steps` as a superset/vendor extension; the standard fields are what make it visible.

Gate verbosity by the existing `diagnostic_mode` / thinking-level config so clients can opt in/out.

## Acceptance criteria

- [ ] OpenAI Chat responses include `reasoning_content` (non-streaming) and stream `delta.reasoning_content` (streaming) carrying the thinking trace.
- [ ] OpenAI Responses stream reasoning summary events.
- [ ] Anthropic `/v1/messages` emits a `thinking` content block (and streamed `thinking_delta`).
- [ ] A thinking-capable CLI (codex reasoning summaries and/or a Claude/DeepSeek-style client) driving Formal AI **visibly displays** the reasoning for a prompt.
- [ ] Behavior respects the diagnostic/thinking-level config.
- [ ] End-to-end tests assert reasoning is present in each protocol's standard field, streaming and non-streaming.

This was found while verifying the issue-#538 premise that Formal AI should be drivable by its own Agent CLI to *"call bash commands"* and complete real tasks.

## Setup

- `@link-assistant/agent` **0.24.0** (updated via `bun add -g @link-assistant/agent@latest`).
- Provider config at `~/.config/link-assistant-agent/opencode.json`: a `formalai` provider using `@ai-sdk/openai-compatible`, `baseURL: http://127.0.0.1:8080/v1`, model `formal-symbolic-production`.
- Server: `formal-ai serve` (v0.254.0, stable rustc 1.96.1).

## Defect 1 â€” streaming hides all model output (see #604)

Agent CLI uses the same `@ai-sdk/openai-compatible` streaming path as opencode, so it hits **#604**: even `agent -p "hi"` returns an **empty** assistant turn. The `step_finish` event shows work happened but no text surfaced:

```json
{"type":"step_finish","part":{"reason":"unknown","tokens":{"input":1,"output":21,"reasoning":0},
 "model":{"providerID":"formalai","requestedModelID":"formal-symbolic-production"}}}
```

Because the streamed payload is `chat.completion` (with `message`) rather than `chat.completion.chunk` (with `delta`), **nothing the model emits â€” including any `tool_calls` â€” reaches the client.** This alone blocks tool use over the Agent CLI. Fixing #604 is a prerequisite.

## Defect 2 â€” the `bash` tool is never granted / never mapped (new)

Tested **server-direct** (bypassing streaming) with `FORMAL_AI_AGENT_MODE=1` so tool execution is enabled. The server still **never emits `tool_calls`** for a shell request. Failing prompts, by advertised tool name:

| Advertised tool | `finish_reason` | `tool_calls` | Server text |
| --- | --- | --- | --- |
| `bash` | `stop` | **NONE** | `Tool calls are not allowed for tool:bash: no installed associative package grants tool:bash.` |
| `shell` | `stop` | **NONE** | `Tool calls are not allowed for tool:shell: no installed associative package grants tool:shell.` |
| `run_command` | `stop` | **NONE** | misroutes to `write_program`: `no template for language the and task list_files` |

Reproduction (agent mode on):

```sh
FORMAL_AI_AGENT_MODE=1 ./target/debug/formal-ai serve &
curl -s http://127.0.0.1:8080/v1/chat/completions -d '{
  "model":"formal-symbolic-production",
  "messages":[{"role":"user","content":"Run the ls command to list files in the current directory."}],
  "tools":[{"type":"function","function":{"name":"bash","description":"Run a bash command",
    "parameters":{"type":"object","properties":{"command":{"type":"string"}},"required":["command"]}}}]
}'
# -> finish_reason: stop, tool_calls: NONE, content: "...no installed associative package grants tool:bash..."
```

### Root causes (in code)

1. **Agent mode is server-env-gated only.** Tool execution requires `FORMAL_AI_AGENT_MODE=1` (`src/solver_helpers.rs:842`). Without it, every tool request is refused with *"Tool calls and function execution are not allowed without explicit agent mode."* There is no per-request way for a CLI to opt in.
2. **No package grants `tool:bash` / `tool:shell`.** `default_associative_packages()` (`src/associative_package.rs:566`) grants `tool:calculator`, `tool:web_search`, `tool:javascript_execution`, `tool:write_program`, `tool:concept_lookup`, `tool:web_fetch`, `tool:write_file`, `tool:run_command` â€” **but not `tool:bash` or `tool:shell`**.
3. **No tool-name â†’ capability mapping.** Every agent CLI (codex, opencode, Agent CLI, claude) advertises a tool literally named **`bash`**. The server looks up capability `tool:bash` verbatim and finds nothing, even though a `tool:run_command` capability exists. Nothing maps the advertised `bash`/`shell` tool name onto `run_command`.

## Proposed fix

- **Map the conventional shell tool names** (`bash`, `shell`) to the shell/run-command capability, so a CLI advertising `bash` triggers the granted command-execution path.
- **Grant a shell/bash capability by default** (in an agent-mode package), or document the exact package to import â€” executed in the existing bounded/isolated agent workspace (honoring `--read-only` / `permission-mode`, per NON-GOALS on bounded autonomy). Reuse the isolated `AgentWorkspace` the agentic-coding loop already uses.
- **Emit real `tool_calls`** (OpenAI `tool_calls` / streamed `delta.tool_calls`) so the CLI executes the command in its own sandbox and feeds the result back â€” the normal agent loop â€” rather than the server refusing or answering in prose.

## Acceptance criteria

- [ ] With #604 fixed, `agent -p "run ls to list files here" --model formalai/formal-symbolic-production` results in a `bash`/shell `tool_calls` with `command: "ls"`, the CLI runs it, and the file list is summarized back.
- [ ] The server emits `tool_calls` (not prose) for a natural-language shell request when a `bash`/`shell` tool is advertised and execution is permitted.
- [ ] `bash` and `shell` tool names resolve to the shell/run-command capability.
- [ ] Shell execution respects `--read-only` / `permission-mode` and runs in the bounded isolated workspace.
- [ ] A per-request / config way to enable agent tool execution exists (not only the `FORMAL_AI_AGENT_MODE` server env), or it is clearly documented.
- [ ] End-to-end test: Agent CLI drives Formal AI to run `ls` and report the listing.

TITLE: with-formal-ai: CLI helper to run/preconfigure codex, opencode, gemini against Formal AI (with -g/--global permanent setup)
CLOSED: 2026-07-03T08:21:45Z
BODY:
## Summary

Add a `with-formal-ai` CLI helper that makes it trivial to run any supported external agent CLI (codex, opencode, gemini, â€¦) against a local Formal AI server â€” by presetting the right environment variables / config per tool â€” and, with `-g`/`--global`, to **permanently** preconfigure those tools to use Formal AI as their default provider.

Today, pointing each CLI at Formal AI requires tool-specific, easy-to-get-wrong config (a `model_providers` TOML block for codex, an `opencode.json` provider for opencode, `GEMINI_API_KEY` + `GOOGLE_GEMINI_BASE_URL` for gemini). `with-formal-ai` should collapse all of that into one command.

## Motivation

Each tool configures a custom provider completely differently:

| Tool | How it targets a custom endpoint |
| --- | --- |
| **codex** | `-c model_providers.<id>.base_url=...`, `wire_api="responses"`, `-c model_provider=...`, `-c model=...` |
| **opencode** | `opencode.json` â†’ `provider.<id>` with `@ai-sdk/openai-compatible`, `options.baseURL`, `models` |
| **gemini** | `GEMINI_API_KEY=<any>` + `GOOGLE_GEMINI_BASE_URL=<base>` (`gemini-api-key` auth); Vertex via `GOOGLE_VERTEX_BASE_URL` |

A user should not have to remember any of this. `with-formal-ai codex "hi"` should just work.

## Proposed UX

### Ephemeral (per-invocation) â€” preset env/config, exec the tool

```sh
with-formal-ai codex "hi"       # runs codex against local Formal AI
with-formal-ai opencode run "hi"
with-formal-ai gemini -p "hi"
```

- Detects/creates the target's provider config, sets the model to the `formal-ai` alias (depends on #605), and `exec`s the tool with all remaining args passed through verbatim.
- Options: `--base-url` (default `http://127.0.0.1:8080`), `--protocol` (openai|gemini|vertex|anthropic, default per tool), `--model` (default `formal-ai`), `--start-server` (spawn `formal-ai serve` if nothing is listening, then run), `--port`.

```sh
with-formal-ai -g codex        # permanently make codex default to Formal AI
with-formal-ai -g opencode
with-formal-ai -g gemini

```

- Writes the provider config into each tool's **persistent** config location:
  - codex â†’ `~/.codex/config.toml` (add `model_providers.formalai`, set `model_provider`/`model`)

  - gemini â†’ shell profile / gemini settings (`GEMINI_API_KEY`, `GOOGLE_GEMINI_BASE_URL`, auth type)
- **Idempotent and reversible:** back up the existing config before writing; `--undo` restores it. Never clobber unrelated settings â€” merge, don't overwrite.
- Print exactly what was changed and where.

=========== ISSUE 605 ===========
TITLE: Model id should be formal-ai (rename): remove user-facing formal-symbolic-production, accept @link-assistant/formal-ai aliases
CLOSED: 2026-07-02T23:31:01Z
BODY:
## Summary

The server currently exposes and accepts exactly one model id â€” `formal-symbolic-production` (`DEFAULT_MODEL`) â€” in both `GET /v1/models` and the `model` field of chat/responses/messages requests. **`formal-symbolic-production` is not the project name and only confuses users. It must be removed.** The canonical model id should become **`formal-ai`**, with a small set of friendly aliases resolving to it.

## Motivation

Every client config in the ecosystem asks the user to type a model name (`codex -c model=...`, `opencode -m <provider>/<model>`, `gemini -m ...`). `formal-symbolic-production` is unmemorable, vendor-internal, and not the product's name â€” it will confuse anyone configuring a client. Users expect to type `formal-ai`. The planned `with-formal-ai` wrapper CLI presets `model=formal-ai`, so `formal-ai` must be the real, canonical id.

## Proposed behavior

### Rename the canonical model id to `formal-ai`

- `DEFAULT_MODEL` becomes **`formal-ai`**.
- **Remove `formal-symbolic-production` entirely** â€” it must no longer be the canonical id, must not be advertised in `GET /v1/models`, and should not be presented anywhere users see model names. (It may, at most, be tolerated as a silent legacy alias for one transition release if truly needed â€” but the default expectation is removal; do not surface it.)

### Accepted request aliases (all resolve to the canonical `formal-ai` model)

The `model` field on `POST /v1/chat/completions`, `/v1/responses`, `/v1/messages` (and the future `/api/<protocol>/...` routes) should accept, case-insensitively:

- `formal-ai` (canonical)
- `@link-assistant/formal-ai`
- `link-assistant/formal-ai`
- `formal-ai-latest` / `latest` (optional convenience)

Unknown model ids should resolve to the canonical model OR return a clear error â€” decide during design â€” but the aliases above must map to it.

### `GET /v1/models` listing

Advertise **`formal-ai` as the primary/default and only user-facing id** (optionally listing the `@link-assistant/formal-ai` scoped form). `formal-symbolic-production` must **not** appear in the listing.

### Alias source is data, not hardcode

Per the project's "data is the interface" principle, the canonical id and alias table should live in seed data (e.g. a `model-aliases` entry) and be read by the server, not baked into a `match` in `server.rs`. Adding an alias should be a data edit.

## Acceptance criteria

- [ ] The canonical model id is `formal-ai`; `DEFAULT_MODEL == "formal-ai"`.
- [ ] `formal-symbolic-production` is **removed** â€” not advertised in `GET /v1/models` and not surfaced to users anywhere.

- [ ] Alias resolution is case-insensitive and driven by seed data, not a hardcoded match.
